# Domain error catalogue

Every `DomainError` class has exactly one i18n key, one stable `code`,
one severity, and one audit projection. The `derivations.ts` helpers
are the single source — this table mirrors them for human readers
(the canonical mapping lives in
`packages/core/src/domain/errors/Errors.ts` and `derivations.ts`).

For the wire-format catalogue (`code` ↔ JSON envelope) consumed by
the Hono router's exhaustive Match, see the auto-generated
[`error-codes.md`](./error-codes.md) regenerated by `just gen-error-docs`.

## Three downstream surfaces

Every error above is consumed by exactly three sinks; the
corresponding helper builds the wire shape for each:

| Sink | Helper | Consumer |
|---|---|---|
| Frontend i18n bundle | `errorToI18nKey(e)` | apps/web (paraglide-js) |
| D1 audit_log row | `errorToAuditEntry(e, ctx)` | `D1AuditLoggerLive` |
| Worker structured log | `toLogPayload(e)` | `WorkersLoggerLive` |

The HTTP envelope (status code + JSON body) is a fourth derived
surface, computed at the API boundary by
`apps/default/src/server/http/errorEnvelope.ts`'s exhaustive
`Match.tagged` over `_tag`. Adding a new error class therefore
touches exactly two files (`Errors.ts` for the class, this table for
the human contract); the four downstream surfaces pick it up
automatically — including the HTTP status mapping, which becomes a
compile error until the new arm is added.
